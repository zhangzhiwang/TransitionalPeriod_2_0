## 1. JDBC 驱动的加载是不是打破了双亲委派模型？

**答：可以这么说。JDBC 的 SPI 自动加载场景，是借助线程上下文类加载器（TCCL）绕过了双亲委派。**

### 双亲委派在这里卡在哪

- `DriverManager` 在 JDK 里，由 Bootstrap / Platform ClassLoader 加载
- MySQL 等驱动在应用 classpath 上，由 AppClassLoader 加载
- 父加载器看不到子加载器的类，所以如果严格只用 `DriverManager` 自己的 ClassLoader 去找 `META-INF/services/java.sql.Driver`，加载不到第三方驱动

### 实际怎么绕过去的

JDK 用了 **线程上下文类加载器（TCCL）**：

1. `DriverManager` 自己不直接用 Bootstrap 去加载驱动实现
2. 改用 `Thread.currentThread().getContextClassLoader()`（一般是 AppClassLoader）
3. 再去发现并加载 SPI 里的驱动实现类

这等于：上层代码反过来用下层 ClassLoader 加载类，所以常被说成“打破双亲委派”。

### 和 `Class.forName` 的关系

- 早期：`Class.forName("com.mysql.cj.jdbc.Driver")` —— 用调用方的 ClassLoader，本身不破坏双亲委派
- JDBC 4.0+：SPI 自动发现（`ServiceLoader` + TCCL）—— 才是面试里常说的那个例外

同类例子还有：JNDI、JAXP、部分框架用 TCCL 加载实现。

## 2. 不用线程上下文类加载器，是不是也能加载到 classpath 下面的驱动实现类？

**答：能。** 不用 TCCL 也可以加载到 classpath 上的驱动，关键是：谁在加载，用哪个 ClassLoader。

### 不用 TCCL 的常见做法

1. 应用代码里显式加载：

```java
Class.forName("com.mysql.cj.jdbc.Driver");
```

调用方是 AppClassLoader 加载的，所以会用 AppClassLoader 去找 jar，根本不需要 TCCL。

2. 手动注册：

```java
DriverManager.registerDriver(new com.mysql.cj.jdbc.Driver());
```

3. 显式传入 ClassLoader：

```java
Class.forName("com.mysql.cj.jdbc.Driver", true, appClassLoader);
// 或
ServiceLoader.load(Driver.class, appClassLoader);
```

### 那为什么还要 TCCL？

问题出在 **SPI 自动发现** 这条路径：

- 发起加载的是 `DriverManager`（JDK 核心类）
- 若它只用自己的 ClassLoader（Bootstrap/Platform），看不到应用 classpath 上的驱动
- 所以 JDK 改用 TCCL（通常是 AppClassLoader）去扫 `META-INF/services`

| 方式 | 能不能加载 classpath 驱动 | 是否依赖 TCCL |
|------|---------------------------|---------------|
| `Class.forName` / 手动 `new Driver()` | 能 | 否 |
| `DriverManager` SPI 自动发现 | 能 | 是（JDK 内部用了 TCCL） |

**结论：** 不是“没有 TCCL 就绝对加载不到”；而是从 JDK 核心类出发自动发现 SPI 时，才必须绕一圈用 TCCL（或显式传入 AppClassLoader）。应用自己去 `Class.forName`，一样能加载。

## 3. DriverManager 不用 TCCL，加载不到位于 classpath 下面的实现类吗？

**答：对，就 SPI 自动发现这条路来说：DriverManager 如果只用自己的 ClassLoader，一般加载不到只在 classpath 上的驱动实现。**

原因：

1. `DriverManager` 是 JDK 核心类，由 Bootstrap / Platform ClassLoader 加载
2. 它若只用“自己的 ClassLoader”去 `Class.forName` / 扫 SPI，找的是启动类路径，不是应用 classpath
3. MySQL 等驱动 jar 在应用 classpath 上，只有 AppClassLoader 能看到

所以父加载器看不见子加载器 classpath 上的类 —— 不是 DriverManager“不会找”，而是它自己的 ClassLoader 权限不够。

### 边界补充

- 你已经用应用代码 `Class.forName` / `registerDriver` 注册过了：连接时不需要再靠 SPI 去加载，这时和 TCCL 无关
- TCCL 为 null 时：有的 JDK 会对 `ServiceLoader` 回退到系统类加载器（多半是 AppClassLoader），那是另一条兜底，不等同于“用 DriverManager 自己的 Bootstrap ClassLoader 也能加载”

**一句话：** 对“DriverManager 自己去发现并加载 classpath 上的 Driver 实现”这件事 —— 不用 TCCL（或不显式传入 AppClassLoader），通常就是加载不到。

## 4. 用户观点辨析：不通过 TCCL，DriverManager 能不能直接拿到 AppClassLoader？

### 用户理由

DriverManager 通过 TCCL 最终拿到的是 AppClassLoader 实例（如果没有设置过上下文加载器的话），然后用它去加载 classpath 下面的类。  
认为不通过 TCCL 也能拿到 AppClassLoader，因为 DriverManager 和 AppClassLoader 本身都在 rt.jar 里、都由根类加载器加载，两者互相可见，所以 DriverManager 可以直接获得 AppClassLoader 的实例。

### 结论

**结论半对，推理不对。**

“不用 TCCL 也能拿到一个能加载应用 classpath 的 ClassLoader”——在纯 SE、普通 `java -jar` / IDE 跑 main 的场景下，这点可以成立。  
但“因为 DriverManager 和 AppClassLoader 都在 rt.jar、互相可见，所以能直接拿到实例”——这个推理不成立。

### 错在哪

1. **类互相可见 ≠ 手里已经有实例**  
   Bootstrap 加载了 `DriverManager` 和 `AppClassLoader` 这两个类，只说明 `DriverManager` 可以调用 ClassLoader 相关 API。  
   它并不会因为“同在 rt.jar”就自动持有那个管理应用 classpath 的 AppClassLoader 实例。  
   要拿实例，仍然得走公开入口，例如：
   - `ClassLoader.getSystemClassLoader()`
   - `Thread.currentThread().getContextClassLoader()`
   - 某个已知类的 `xxx.getClassLoader()`

2. **“直接获得”其实就是调用 API，不是特权旁路**  
   SE 场景下用 `getSystemClassLoader()`，和用默认的 TCCL，经常拿到的是同一个 AppClassLoader。  
   所以不是“没法不用 TCCL”，而是 JDK 选了 TCCL 这条路径，不是“除了 TCCL 就拿不到”。

### 为什么要专门用 TCCL？

因为目标环境经常不是“整个 JVM 只有一个应用 ClassLoader”：

| 环境 | `getSystemClassLoader()` | 默认 / 容器设置的 TCCL |
|------|--------------------------|-------------------------|
| 普通 main 程序 | 通常就是 AppClassLoader，能看到 classpath 驱动 | 默认也常是它，效果类似 |
| Tomcat / Java EE 等 | 多半只能看到容器本身 | 常是 WebAppClassLoader，能看到 `WEB-INF/lib` 里的驱动 |

容器里 JDBC 驱动往往在子 ClassLoader 上。这时：

- 只用 system ClassLoader → 可能仍加载不到
- 用 TCCL → 才能跟当前线程/当前应用对上

这才是 DriverManager / SPI 用 TCCL 的真正原因：可插拔、按线程切换正确的加载器，而不只是“绕开双亲委派拿 AppClassLoader”。

### 更准确的说法

- 对的部分：不用 TCCL，在 SE 下用 `ClassLoader.getSystemClassLoader()`，确实也能加载 classpath 上的驱动；默认没改过 TCCL 时，两者常常是同一个
- 不对的部分：不是因为都在 rt.jar 就“互相可见所以直接有实例”；ClassLoader 实例必须显式获取
- 面试口径：JDBC SPI 用 TCCL，是为了让 Bootstrap/Platform 上的 `DriverManager` 委托合适的下层 ClassLoader 去加载实现；在容器里这比死盯 system ClassLoader 更重要

## 5. Tomcat 环境下，为什么 `getSystemClassLoader()` 加载不到 classpath 下的驱动？

核心点：Tomcat 里“classpath”分好几层。  
`getSystemClassLoader()` 拿到的是 JVM 启动时的系统 ClassLoader，它只看 Tomcat 进程启动时的那份 classpath，看不到放在各 Web 应用 `WEB-INF/lib` 里的 MySQL 驱动。

### Tomcat 类加载大致层级

```text
Bootstrap ClassLoader          （JDK 核心）
        ↑
System / AppClassLoader        （启动 Tomcat 时的 CLASSPATH）
        ↑
Common ClassLoader             （$CATALINA_HOME/lib，如 servlet-api、catalina）
        ↑
Webapp ClassLoader             （某个 WAR：WEB-INF/classes + WEB-INF/lib）
```

常见部署是：`mysql-connector-j.jar` 放在 `WEB-INF/lib` → 由 WebappClassLoader 负责。

### 为什么加载不到

1. Tomcat 启动时，System ClassLoader 的 classpath 通常只有启动相关 jar（如 `bootstrap.jar`），不包含某个 WAR 的 `WEB-INF/lib`
2. 双亲委派下，父看不到子：System 是 Webapp 的祖先，Webapp 里有的类，System 查不到、也加载不了
3. 因此 `DriverManager` 若用 `ClassLoader.getSystemClassLoader()` 去加载 `com.mysql.cj.jdbc.Driver`，会在 System 这一层失败 —— 不是驱动不在“项目依赖”里，而是不在 System 能看到的路径里

### 和 TCCL 的对比

| 获取方式 | Tomcat 请求线程里通常是谁 | 能否看到 `WEB-INF/lib` 里的 MySQL |
|----------|---------------------------|-----------------------------------|
| `ClassLoader.getSystemClassLoader()` | System / AppClassLoader | 一般不能 |
| `Thread.currentThread().getContextClassLoader()` | WebappClassLoader（容器会设） | 能 |

Tomcat 处理请求时，会把当前线程的 TCCL 设成该 Web 应用的 ClassLoader。  
`DriverManager` / SPI 走 TCCL，才能落到 WebappClassLoader，进而加载到 `WEB-INF/lib` 里的驱动。

### 容易混淆的一点

如果说“放在 Tomcat 公共 lib 里也能被加载”——那是因为 jar 在 Common ClassLoader（`$CATALINA_HOME/lib`）上，跟 System 还不是一回事；而且所有 Web 应用会共享这份驱动。  
日常面试说的“Tomcat 下 System ClassLoader 加载不到 MySQL”，默认场景就是：驱动在 WAR 的 `WEB-INF/lib`，而不在 JVM/System classpath 上。

**一句话：** `getSystemClassLoader()` 拿到的不是“当前 Web 应用的 ClassLoader”，所以看不到该应用 classpath 上的驱动实现类。

## 6. AppClassLoader 的实例是单例的吗？

**答：在标准 JDK 里，可以认为 AppClassLoader 的实例是单例的。**

更准确地说：

1. JVM 启动时，`Launcher` 只会创建 **一个** 应用类加载器实例
2. `ClassLoader.getSystemClassLoader()` 每次返回的都是 **同一个对象**
3. 整个 JVM 进程里，系统类加载器通常就这一个

```java
ClassLoader c1 = ClassLoader.getSystemClassLoader();
ClassLoader c2 = ClassLoader.getSystemClassLoader();
// c1 == c2  → true
```

### 要注意

1. **单例指的是“系统 AppClassLoader”**  
   你自己 `new` 出来的 `URLClassLoader`、Tomcat 的 `WebappClassLoader` 等，都是别的实例，不是 AppClassLoader
2. **Java 9+ 模块化后名字变了**  
   内部实现常是 `jdk.internal.loader.ClassLoaders$AppClassLoader`，但语义不变：系统类加载器仍然只有一个
3. **“单例”不等于“所有应用类都由它加载”**  
   在 Tomcat 里，WAR 里的类往往由 WebappClassLoader 加载；AppClassLoader 仍是单例，但不再负责那些类

**一句话：** AppClassLoader（系统类加载器）在单个 JVM 里是单例；但 JVM 里可以有无数个别的 ClassLoader 实例。

## 7. 所有线程获得的 AppClassLoader 实例都是同一个吗？

**答：对。** 在同一个 JVM 里，所有线程通过 `ClassLoader.getSystemClassLoader()` 拿到的 AppClassLoader（系统类加载器）都是同一个实例。

不过别和 **线程上下文类加载器（TCCL）** 搞混：

| API | 是否所有线程同一个 |
|-----|-------------------|
| `ClassLoader.getSystemClassLoader()` | 是，全局单例 |
| `Thread.currentThread().getContextClassLoader()` | 不一定，每个线程可各自设置 |

所以：AppClassLoader 全 JVM 共享一份；TCCL 可能不同线程不同，Tomcat 就是这么给每个请求线程挂上对应 Web 应用 ClassLoader 的。

## 8. AppClassLoader 实例是什么时候创建的？

**答：在 JVM 启动阶段创建，且早于你的 `main` 方法执行。**

### 时机（按启动顺序）

1. 创建 Bootstrap ClassLoader（本质是 JVM 原生部分，不是普通 Java 对象）
2. 创建 ExtClassLoader（Java 8）或 Platform ClassLoader（Java 9+）
3. 创建 AppClassLoader，并设为系统类加载器
4. 用 AppClassLoader 加载你的主类，再调用 `main`

所以 AppClassLoader 是进程刚启动、进 `main` 之前就建好的，之后 `getSystemClassLoader()` 一直返回这一份。

### 谁创建

- Java 8：`sun.misc.Launcher` 初始化时 `new AppClassLoader(...)`
- Java 9+：`jdk.internal.loader.ClassLoaders` 在启动时创建对应的 App ClassLoader

创建时会把 `java.class.path`（classpath）设进去，之后这个实例就固定了。

**一句话：** JVM 启动初始化类加载体系时创建，全进程一份，`main` 开始前就已存在。

## 9. 可以自己创建一个 AppClassLoader 实例吗？

**答：一般不行。** 你拿到的系统 AppClassLoader 是 JVM 启动时内部创建的那一个；业务代码通常也不能随便 `new` 出一个官方同款。

### 为什么不行

1. **实现类在 JDK 内部**
   - Java 8：`sun.misc.Launcher$AppClassLoader`
   - Java 9+：`jdk.internal.loader.ClassLoaders$AppClassLoader`  
   Java 9+ 模块强封装，普通代码访问不了这些类
2. **构造被限制**  
   就算用反射硬刚，构造器也不对外公开，且系统类加载器入口只认启动时注册的那份
3. **就算造出来也没用**  
   `ClassLoader.getSystemClassLoader()` 不会改成指向你 new 的那个；全 JVM 仍用启动时那一个

### 你真正能做的

| 需求 | 做法 |
|------|------|
| 自定义从某些 jar/路径加载 | `new URLClassLoader(urls, parent)` |
| 或自己继承 `ClassLoader` / `SecureClassLoader` | 自定义 `findClass` |
| 替换系统类加载器（极少用） | 启动参数 `-Djava.system.class.loader=你的类`（老机制，一般不推荐） |

**一句话：** 不能自己再造一个“系统 AppClassLoader”；可以自己建功能类似的自定义 ClassLoader。
